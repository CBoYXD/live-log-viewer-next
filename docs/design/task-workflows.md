# Task workflows: задача → етапи → імплементери → review — DRAFT (pre-grill)

Status: **first draft, not yet grilled.** Кожен «HARD DECISION» — відкрите
питання інтерв'ю; рекомендація — авторська, не погоджена.

## Problem

Сьогодні review-flow чіпляється до вже запущеної вручну сесії: користувач сам
створює worktree, сам спавнить імплементера, сам пише йому задачу, і тільки
потім вмикає цикл review. Хочеться «запустив задачу — і воно поїхало»: готовий
шаблон (наприклад, «UI пише Fable, бекенд — Codex»), окремий worktree
створюється сам, імплементери працюють паралельно або послідовно, після них
автоматично стартує звичний review-цикл.

## Existing machinery to reuse (verified in code)

- `src/lib/flows/*` — стейт-машина одного review-циклу: маркер
  `REVIEW_READY:` → свіжий headless/pane ревʼюер → вердикт → relay → fixing.
  Тікає на полінгу `/api/files`, стан у `~/.claude/viewer-state/flows/flows.json`,
  переживає рестарти (pid/pane re-attach).
- `POST /api/spawn` + `freshSpecFor()` — спавн агента в tmux з моделлю/effort,
  transcript відомий наперед (claude), lineage у handoff-lineage.json.
- `sendToImplementer()` — доставка тексту в живу панель або через resume.
- `detectReadyMarker()` — «маркер лише після закритого ходу» — готовий шаблон
  для будь-яких етапних маркерів.
- `FlowPreset` — іменовані пари ролей; природно розширюється до шаблонів
  workflow.

## Proposed model (draft)

```
Workflow = задача + worktree + послідовність етапів (stages)
Stage    = implement { agents: Role[] (паралельно) } | review { reviewer: Role }
Role     = { name, engine, model, effort, brief }   // brief: "You own the UI…"
```

- Етапи виконуються послідовно; всередині implement-етапу агенти працюють
  паралельно в одному worktree.
- `review`-етап делегує наявній стейт-машині Flow (implementer циклу — один
  призначений «lead» імплементер).
- Стан у `workflows.json` поруч із `flows.json`, той самий tick-патерн.

## HARD DECISIONS (усі — на гриль)

**W1 — шар чи переписування.** Workflow — окремий оркестратор *над* flows
(review-етап створює звичайний Flow і чекає його термінального стану) чи
розширення самої Flow-машини новими станами?
*Рекомендація:* шар над flows. Flow-машина вивірена (рестарти, re-attach,
вердикти) — вбудовування етапів усередину подвоїть її складність; workflow
лише спавнить, чекає маркерів і створює Flow.

**W2 — worktree.** Хто і де створює? Одна на workflow чи на імплементера?
*Рекомендація:* одна на workflow: `git worktree add -b <branch>` від HEAD
основного чекаута проєкту, у стандартному місці (`<repo>-wt/<slug>` або
конфігурований корінь). Спільний diff — це те, що ревʼюїться; окремі worktree
на агента потребують merge-механіки, якої у v1 нема.

**W3 — паралельність у одному worktree.** Два агенти в одному дереві можуть
конфліктувати (git index, спільні файли).
*Рекомендація:* дозволити паралель лише коли шаблон явно ділить володіння
(brief кожної ролі описує зону: «only src/components/**»), і заборонити
git-операції всім, крім lead (інші комітять через lead або не комітять
взагалі — вирішити). Альтернатива: послідовно за замовчуванням.

**W4 — завершення implement-етапу.** Як дізнатися, що агент закінчив етап?
*Рекомендація:* маркер `STAGE_DONE: <note>` тим самим механізмом, що
`REVIEW_READY` (лише після закритого ходу). Етап завершено, коли всі агенти
етапу поставили маркер.

**W5 — кому летять findings у review-циклі при кількох імплементерах.**
*Рекомендація:* lead-імплементер (перша роль шаблону) — він і є implementer
створеного Flow; решта агентів етапу на цей час уже вільні. Роутити findings
по ownership-глобах — v2.

**W6 — шаблони.** Де живуть і як виглядають?
*Рекомендація:* `workflow-templates.json` у viewer-state, сідовані дефолти
(«solo+review», «ui-fable + backend-codex + review»), редагування через файл
у v1, UI-редактор пізніше.

**W7 — вхід задачі.** Звідки береться текст задачі і де він живе?
*Рекомендація:* діалог «Запустити задачу» (проєкт → шаблон → текст задачі) +
`POST /api/workflows`; текст персиститься в записі workflow і вставляється в
перший промпт кожної ролі разом з role brief.

**W8 — завершення workflow.** Що після approved?
*Рекомендація:* workflow стає `done`, worktree і гілка лишаються — merge/PR
руками користувача; авто-PR out of scope.

**W9 — UI.** Як workflow видно на схемі?
*Рекомендація:* усі транскрипти workflow анотуються (як FlowAnnotation) і
групуються на дошці; стрічка стану workflow — розширення наявної flow strip.

**W10 — рестарти/збої.** Агент етапу помер, маркер не приходить.
*Рекомендація:* потік needs_decision, як у flows: жодних авто-ребутів у v1;
users бачить стан на стрічці і чинить руками (retry stage / kill).

## Out of scope (draft)

- DAG-етапи (лише лінійний список).
- Merge кількох worktree, авто-PR, авто-merge.
- Роутинг findings по ownership.
- UI-редактор шаблонів.
